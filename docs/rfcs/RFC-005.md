## 2. Objetivo

Desenvolver uma estrutura mínima do Backend da TechStore, estabelecendo uma base de projeto organizada, testável e alinhada às práticas de CI/CD e arquitetura já adotadas pela empresa, servindo como ponto de partida para o desenvolvimento das futuras funcionalidades como: Login, produtos, pagamentos, etc.

---

## 3. Descrição da Mudança

Será adicionada ao repositório a estrutura inicial de um serviço backend em NODE.js + TypeScript + Express, organizada por responsabilidade (rotas, controllers, middlewares, configuração), já preparada para receber novos módulos sem necessidade de reestruturação futura.

---

## 4. Justificativa Técnica

Separar rotas, controllers e middlewares reduz acoplamentos e facilita a adição de novos módulos (login, produtos, pagamentos) sem reescrever a base.

O endpoint (/health) viabiliza checagens automatizadas de disponibilidade, essenciais para o pipeline de CI/CD já implementado.

TypeScript reduz erros em tempo de desenvolvimento e melhora a manutenibilidade em projeto com múltiplas com equipes trabalhando em pararelo (alinhado à decisão de microsserviços da RFC da arquitetura)

A estrutura segue convenções amplamento adotadas no mercado, reduzindo a curva de aprendizado para novos integrantes da equipe.

---

## 5. Impacto Esperado

(x) Baixo

( ) Médio

( ) Alto

Como se trata da criação da estrutura inicial do backend, o impacto no sistema em produção é baixo, pois ainda não há código legado ou usuários ativos nessa nova estrutura. O impacto para a equipe de desenvolvimento é "Altamente positivo", pois estabelece um padrão de organização de pastas que evita retrabalho e facilita a integração paralela de novas funcionalidades.

---

## 6. Riscos

- Conflito de Dependências - Risco de incompatibilidade de versões entre pacotes essenciais.
- Configuração de Ambiente - Membros da equipe enfrentarem dificuldades para rodar o projeto localmente devido a diferentes versões do Node.js instaladas em suas máquinas.
- Curva de Aprendizado - Desenvolvedores menos familiarizados com TypeScript ou com a arquitetura de pastas proposta levarem mais tempo para iniciar suas tarefas.

---

## 7. Plano de Rollback

Como se trata da criação de uma estrutura nova, isolada em sua própria branch e Pull Request, o rollback é simples: reverter o merge (git revert) ou simplesmente não mesclar o PR. Nenhum sistema em produção depende dessa estrutura até que ela seja explicitamente adotada por uma funcionalidade.

## 8. Critérios de Validação

- O comando npm install && npm run dev sobe o servidor localmente sem erros.
- O endpoint GET /health responde 200 OK.
- O comando npm test executa o teste de exemplo com sucesso.
- Estrutura de pasta revisada e aprovada por pelo menos dois membros da equipe no Pull Request.
